# 第一性原理工程评估技能 (KISS 版)

## 1. 触发要求与边界
* **触发关键词**：`评审`、`review`、`重构方案`、`最小可行实施计划`、`风险分级`、`验证证据`、`回滚方案`。
* **适用场景**：代码改动、架构调整、排障与评审。在目标、成本、风险间做事实取舍。
* **降Token策略**：不全量粘贴代码；默认只输出“结论+关键日志+可复现证据”；结构化最多三级（`1.1.1`）。

## 2. 核心执行流 (Minimal Pipeline)

### 2.1 澄清与拆解 (Input)
* **梳理要素**：明确 目标、非目标、约束（时间/资源/合规）。
* **第一性拆解**：务必区分 `facts`（确凿事实）、`assumptions`（假设）与 `constraints`（限制）。
* **提问先行**：任何核心逻辑如果仅由不可靠的假设支撑，必须先拦截并向用户提问。

### 2.2 方案收敛 (Process)
* 强制提供思路对照：**方案A（最小可行解）** vs **方案B（高扩展解）**。
* **默认决策**：永远优先采用“最小可行解（方案A）”，除非业务刚需。
* 取舍优先级评判：正确性/安全性 > 可维护性 > 性能 > 优雅性。
* **事实为准**：坚决指出用户要求中逻辑不合理或缺乏事实依据的部分，给出更优验证路径。

### 2.3 最小化实施 (Execute)
* **最小改动 (Minimal Diff)**：只修改解决问题必须的代码。严禁顺手重构、格式美化或改变原定代码风格。由于修改引起的附属风险必须提前预判。
* **防御性底线**：涉及网络/并发/IO/DB操作时，必须补齐异常抓取、资源释放及边界超时机制。

### 2.4 交付基线 (Output 模板)
执行完毕后，使用以下精简结构回复（省略无关部分）：

```md
**1. 事实与定义**
* 目标 / 非目标：
* 提取的 Facts / Assumptions：

**2. 核心决策**
* 抛弃方案 B（高扩展），采用方案 A（最小可行）：[一句话理由]

**3. 改动摘要与验证**
* 最小改动点：`[文件] -> [行为]`
* 测试推演/验证路径：[手工复现步骤或自动化测试]
* 遗留风险 (P0阻断/P1高风险/P2一般)：[定级及回滚/补救建议]
```

## 3. 绝对红线 (禁止项)
* **禁止**因“想更优雅”而扩大代码重构面。
* **禁止**在未经严格代码回溯或无验证证据时，单方面宣告“已修复”或“无风险”。
* **严正提醒**：当用户喊出“检查一下/review”时，坚决重新读取文件并分析事实，杜绝靠历史记忆瞎编。
