# 工程验证清单（First Principles）

用于补充 `SKILL.md` 的“验证与交付”步骤。  
在任务涉及代码改动、架构调整、性能优化、线上问题排查时按需选用。

## 1. 使用规则
1.1 本清单用于在有限时间内形成可复核的验证证据，而不是机械执行全部条目。  
1.1.1 [ ] 先选最相关的 1-2 组清单。  
1.1.2 [ ] 每条清单记录结果：`通过 / 不通过 / 不适用`。  
1.1.3 [ ] 对“不通过”项补充风险和处置计划。  

## 2. 正确性与回归
2.1 本组用于确认改动后功能行为正确且未引入明显回归。  
2.1.1 [ ] 需求目标与非目标已明确。  
2.1.2 [ ] 关键路径输入输出与预期一致。  
2.1.3 [ ] 边界条件（空值、异常值、极端值）已覆盖。  
2.1.4 [ ] 失败路径行为可预期（错误码/报错信息清晰）。  
2.1.5 [ ] 原有功能未出现明显回归。  

## 3. 安全与边界防护
3.1 本组用于确认外部输入、异常路径与敏感信息处理均处于受控状态。  
3.1.1 [ ] 外部输入做了类型和范围校验。  
3.1.2 [ ] 文件/网络/数据库操作具备异常处理。  
3.1.3 [ ] 超时和重试策略可解释且有上限。  
3.1.4 [ ] 敏感信息未写入日志。  

## 4. 性能与资源
4.1 本组用于确认关键路径性能与资源使用不会造成可预见的容量风险。  
4.1.1 [ ] 热路径不存在明显重复计算或不必要 I/O。  
4.1.2 [ ] 关键环节复杂度在可接受范围。  
4.1.3 [ ] 并发/异步处理没有资源泄漏风险。  
4.1.4 [ ] 高开销操作有降级或兜底策略。  

## 5. 可维护性与可观测
5.1 本组用于确认改动可长期维护，且出现问题时可定位可追踪。  
5.1.1 [ ] 改动范围最小化，无无关重构。  
5.1.2 [ ] 命名、结构、风格与现有代码一致。  
5.1.3 [ ] 关键分支有必要注释，非显而易见逻辑可读。  
5.1.4 [ ] 错误日志包含定位上下文（模块、输入摘要、异常类型）。  

## 6. 发布与回滚
6.1 本组用于确认发布前置条件与失败回退路径均已明确并可执行。  
6.1.1 [ ] 变更影响范围已识别（模块、接口、上下游）。  
6.1.2 [ ] 发布前置条件明确（配置、依赖、数据准备）。  
6.1.3 [ ] 回滚条件与步骤明确。  
6.1.4 [ ] 风险等级已标注（P0/P1/P2）。  

## 7. 交付记录模板
```md
1. 验证结论
1.1 结论状态：通过 / 有条件通过 / 不通过

2.2 安全与边界防护：
2.3 性能与资源：
2.4 可维护性与可观测：
2.5 发布与回滚：

3. 未通过项与处置
3.1 项目：
3.2 风险等级：
3.3 处置动作：
```
